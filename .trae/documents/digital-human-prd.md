# MetaHuman 交互 Demo/SDK PRD

## 1. 产品定义

MetaHuman 是一个面向演示、验证与二次开发的数字人交互 Demo/SDK。

它的核心价值不是“做一个完整数字人平台”，而是以尽量低的接入成本，跑通以下交互闭环：

- 3D 数字人渲染与基础动作/表情驱动
- 文本输入、语音输入、语音播报
- 摄像头视觉镜像与基础状态映射
- 后端 LLM 对话生成与前端表现联动
- 无云端配置时仍可稳定演示的 Mock 回退

一句话定位：

> 一个可运行、可展示、可二开、可本地自洽的数字人交互样例工程。

## 2. 目标用户

### 2.1 核心用户
- 需要快速展示“数字人 + 对话 + 语音 + 视觉”能力的售前/方案团队
- 需要基于现有项目做业务接入和功能扩展的前后端开发团队
- 需要验证数字人交互链路可行性的 PoC / Demo 团队

### 2.2 使用诉求
- 能在本地快速跑起来，形成可演示闭环
- 能清楚看懂前后端职责边界与关键模块入口
- 能在不改整体架构的前提下替换模型、页面或业务逻辑
- 能在无 OpenAI Key 时继续完成 UI 演示与前端联调

## 3. 产品边界

### 3.1 当前要解决的问题
- 如何低成本展示数字人交互能力
- 如何把“输入 → 理解 → 回复 → 表现 → 播报”串成一条稳定链路
- 如何让浏览器侧能力（TTS / ASR / MediaPipe）与服务端对话能力配合工作

### 3.2 当前明确不做
- 用户注册 / 登录 / 权限管理
- 模型资源管理后台（上传、审核、发布、版本管理）
- 复杂行为编排系统（时间轴、规则引擎、剧情树）
- 平台级部署管理、多租户、运营后台
- 生产级消息总线、任务调度、监控告警体系

这份边界很重要：

> 文档、代码、演示口径都应坚持“Demo/SDK”定位，避免把仓库描述成完整平台。

## 4. 成功标准

当前版本的成功，不看平台化指标，主要看下面 5 件事是否成立：

1. 页面可正常打开，数字人可正常渲染
2. 文本输入后，能获得结构化回复并驱动数字人反馈
3. 语音输入可进入同一条对话链路
4. 摄像头视觉镜像可得到基础识别结果并映射到表现层
5. 未配置云端 Key 或请求异常时，系统仍能通过 Mock 保持演示可用

## 5. 核心场景与关键路径

### 5.1 场景 A：文本对话驱动数字人

目标：验证最基础、最稳定的交互闭环。

路径：
1. 用户打开页面（默认进入 Advanced 页面）
2. 输入文本并发送
3. 前端请求后端 `POST /v1/chat`
4. 后端返回 `{ replyText, emotion, action }`
5. 前端将结果同步到聊天区、数字人引擎与 TTS
6. 用户看到表情/动作变化，并在未静音时听到语音回复

验收结果：
- 聊天内容可见
- 数字人状态有变化
- 语音播报与文本回复一致
- 出错时用户能得到明确反馈，且页面不崩溃

### 5.2 场景 B：语音输入驱动对话

目标：验证浏览器语音能力与后端对话能力的联动。

路径：
1. 用户点击录音
2. ASR 识别出文本
3. 识别结果进入与文本输入相同的对话链路
4. 返回结果继续驱动表情、动作与播报

验收结果：
- 录音状态可见
- 识别文本可进入聊天流
- 与文本输入路径行为一致

### 5.3 场景 C：视觉镜像驱动表现变化

目标：验证视觉能力可以成为数字人的另一种输入源。

路径：
1. 用户授权摄像头
2. 前端运行 MediaPipe 推理
3. 视觉结果映射为简化 emotion / head action
4. 页面或引擎消费映射结果并更新数字人表现

验收结果：
- 摄像头画面可见
- 识别状态有反馈
- 映射结果能体现在数字人表现层或面板状态中

## 6. 功能范围（以当前实现为准）

### 6.1 数字人渲染
- 基于 Three.js + React Three Fiber 的 3D 渲染
- 支持 GLB / GLTF 模型加载
- 模型缺失或加载失败时回退内置 avatar
- 支持基础播放、重置、自动旋转等演示能力

### 6.2 对话能力
- 前端统一通过 `/v1/chat` 发起请求
- 返回结构固定为：`replyText`、`emotion`、`action`
- 后端支持会话上下文管理
- 配置 OpenAI Key 时调用 LLM
- 无 Key、超时、HTTP 错误、请求异常时自动回退智能 Mock

### 6.3 语音能力
- TTS：将回复文本播报给用户
- ASR：将用户语音转换为文本
- 支持静音、录音、说话等状态同步
- 依赖浏览器 Web Speech API，能力受运行环境影响

### 6.4 视觉能力
- 摄像头画面采集
- MediaPipe Face Mesh / Pose 推理
- 视觉结果映射为简化的情绪与动作信号
- 结果用于演示数字人与用户状态联动的可能性

### 6.5 页面与交互
- `/`、`/advanced`：高级页面，能力最完整
- `/digital-human`：简化页面，适合最小演示
- 支持设置、快捷控制、面板切换、连接状态提示等体验能力

## 7. 非功能要求

### 7.1 可运行性
- 前端可独立启动
- 后端可独立启动
- 默认本地开发环境可完成联调

### 7.2 可降级性
- 后端 LLM 不可用时不阻塞整条演示链路
- 模型加载失败时不直接导致页面不可用
- 视觉 / 语音能力缺失时，其他能力应尽量保持可用

### 7.3 可读性与可扩展性
- 目录结构应能反映模块职责
- 对话、语音、视觉、数字人驱动应相对解耦
- 页面层尽量调用高层服务/编排层，而不是直接堆底层逻辑

## 8. 关键约束

- API Key 仅保存在后端环境变量中，前端不持有密钥
- 浏览器能力依赖用户授权与运行环境，不保证跨设备完全一致
- 当前会话历史以内存形式维护，不适合作为生产级持久化方案
- 当前仓库优先保证演示稳定和代码清晰，而不是追求平台复杂度

## 9. 配置说明

### 9.1 前端
- `VITE_API_BASE_URL`：后端地址，默认 `http://localhost:8000`

### 9.2 后端
- `OPENAI_API_KEY`：可选；不配置则走 Mock
- `OPENAI_MODEL`：可选；LLM 模型名
- `OPENAI_BASE_URL`：可选；支持标准 OpenAI 地址或兼容网关
- `CORS_ALLOW_ORIGINS`：可选；允许跨域来源列表
- `LLM_PROVIDER`：当前保留扩展位，现阶段实际仍以 OpenAI 兼容接口为主

## 10. 后续演进建议

建议沿着“Demo/SDK → 场景化组件 → 平台化能力”的顺序演进，而不是直接跨到平台化：

### P1：增强 Demo 可用性
- 补充更稳定的会话管理与错误提示
- 完善空状态、权限状态、降级状态展示
- 优化多输入源（文本 / 语音 / 视觉）之间的状态协同

### P2：增强 SDK 化能力
- 抽离更清晰的适配层与接口层
- 支持更灵活的模型驱动与动作映射策略
- 形成更明确的二次开发扩展点

### P3：再考虑平台化
- 账号体系
- 模型资产管理
- 行为编排后台
- 多租户与运营能力

在此之前，不建议把仓库文档写成“完整平台产品方案”。

## 11. 最终结论

这个项目最适合被定义为：

> 一个聚焦数字人交互闭环验证的 Web Demo/SDK，而不是完整的数字人平台。

只要文档、代码、页面和对外介绍都围绕这句话保持一致，项目的可信度、可维护性和可演示性就会明显提升。
